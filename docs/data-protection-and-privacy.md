# Data Protection and Privacy Policy

## 1. Overview
This document defines the data protection and privacy practices for the Chioma platform. It outlines how user data is collected, processed, stored, protected, and deleted in alignment with industry standards and regulatory expectations.

The platform may process personal data related to housing services, including user identification, communication, and property-related information.

---

## 2. GDPR Compliance

The platform aligns with core GDPR principles:

- **Lawfulness, Fairness, Transparency**  
  Data must be processed in a lawful and transparent manner.

- **Purpose Limitation**  
  Data is collected only for specific, legitimate purposes (e.g., account creation, housing services).

- **Data Minimisation**  
  Only necessary data is collected.

- **Accuracy**  
  Data must be kept up-to-date and corrected when necessary.

- **Storage Limitation**  
  Data is retained only as long as needed.

- **Integrity and Confidentiality**  
  Data must be protected against unauthorized access or breaches.

- **Accountability**  
  The platform must demonstrate compliance with these principles.

---

## 3. Data Classification

Data is categorized into:

### Public Data
- Non-sensitive, publicly available information

### Internal Data
- System logs
- Operational data

### Sensitive Data
- Full name
- Email address
- Phone number
- Residential or housing-related information
- Uploaded identification or documents

Sensitive data requires stricter handling, access control, and encryption.

---

## 4. Data Collection and Usage

Data collected may include:

- User identity information (name, email)
- Contact details
- Property or housing-related data
- Authentication credentials

Data is used for:

- Account creation and management
- Service delivery (housing-related operations)
- Communication with users
- System monitoring and improvement

---

## 5. Data Retention Policy

- Personal data is retained only for as long as necessary.
- Retention periods should be defined based on:
  - Business needs
  - Legal obligations
- Periodic reviews should be conducted to remove outdated or unnecessary data.
- Data no longer required should be:
  - Deleted, or
  - Anonymised

---

## 6. Data Deletion Policy

- Users may request deletion of their personal data.
- Requests must be:
  - Verified
  - Logged
  - Processed within a defined timeframe

Exceptions:
- Legal or compliance requirements may require retention.

Deletion methods:
- Permanent removal from database
- Secure deletion practices to prevent recovery

---

## 7. Privacy by Design

Privacy must be integrated into system development:

- Collect only necessary data
- Default to minimal data exposure
- Avoid storing sensitive data unless required
- Ensure secure architecture from the start

---

## 8. Access Controls

- Role-based access control (RBAC) must be implemented.
- Access to sensitive data is restricted to authorized personnel only.
- All access should follow the **principle of least privilege**.

Examples:
- Admins: full access
- Staff: limited operational access
- Users: access only to their own data

---

## 9. Encryption Standards

Sensitive data must be protected using encryption:

- **In Transit:** HTTPS (TLS)
- **At Rest:** Database encryption where applicable
- Passwords must be hashed using secure algorithms (e.g., bcrypt)

---

## 10. Audit Logging

The system must log:

- User login attempts
- Data access events
- Data modifications
- Administrative actions

Logs should include:
- Timestamp
- User ID
- Action performed

Logs must be:
- Secure
- Tamper-resistant
- Reviewed periodically

---

## 11. Incident Response

In case of a data breach:

1. Identify the breach
2. Contain the issue
3. Assess impact
4. Notify relevant stakeholders if required
5. Document the incident
6. Implement corrective measures

---

## 12. Compliance

The platform should align with:

- GDPR principles
- Data protection best practices
- Internal security policies

Regular reviews and updates should be conducted to maintain compliance.
